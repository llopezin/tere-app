# ClinicFlow — Client App

React 19 + Vite + TanStack Router + Tailwind v4.

## Getting started

```bash
pnpm install
pnpm --filter @fisio-app/client-app dev
```

## Folder structure

```
src/
├── components/
│   └── ui/                  ← Shared design system primitives (Button, Card, etc.)
│       └── index.ts         ← barrel export
├── routes/                  ← TanStack Router file-based routes
│   ├── __root.tsx           ← root layout
│   ├── index.tsx            ← /
│   └── patient/
│       ├── welcome.lazy.tsx ← page component
│       └── -components/     ← colocated, page-specific components
├── lib/                     ← utilities, auth client, API client
├── index.css                ← Tailwind config + design tokens (@theme)
├── routeTree.gen.ts         ← auto-generated by TanStack Router plugin
└── main.tsx                 ← app entry point
```

### Rules

- **`components/ui/`** — Shared across multiple pages. If only one page uses it, it stays colocated.
- **`routes/<page>/-components/`** — Page-specific components. The `-` prefix tells TanStack Router to ignore the folder. Components migrate to `ui/` when reused elsewhere.
- **`lib/`** — Framework-agnostic utilities (`cn.ts`, future auth client, API client).

## Design tokens

All design tokens live in `src/index.css` inside the `@theme` block. Use Tailwind utility classes referencing these tokens:

```tsx
<div className="bg-primary-600 text-text-inverted rounded-lg shadow-card">
```

When adding a new color/token observed in Figma, **add it to the `@theme` block** — do not use arbitrary values in components.

## String management

This app is **single-language (Spanish)**. Strings are kept **inline in JSX** for simplicity:

```tsx
<h1>Bienvenido</h1>
<p>Gestiona tus citas médicas de forma fácil y rápida</p>
```

For data-driven lists, use a local `const` array within the component file and `.map()` over it:

```tsx
const FEATURES = [
  { title: "Reserva en minutos", description: "Elige tu profesional..." },
  // ...
] as const;

{FEATURES.map((f) => <FeatureItem key={f.title} {...f} />)}
```

**Why not i18n?** Single language, ~16 screens, 1-2 developers. The overhead of key management, message extraction, and runtime lookups provides zero benefit. If a second language is ever needed, inline strings can be migrated to an i18n library at that point.
